## 0. Принцип №1 — двигатель вперёд витрины

Два продукта в одном, не путать:
- **Двигатель (ценность):** автономно делает реальную работу по твоему портфелю и отчитывается.
- **Витрина (вау):** 3D-офис, агенты за столами / на диване / в митинг-руме.

**Правило:** строим двигатель так, чтобы он *излучал события состояния*; 3D-офис —
живое зеркало этих событий, не бутафория. Витрину делаем ПОСЛЕДНЕЙ. Нарушение
этого порядка = красивая пустышка.

## 3. ФАЗЫ (детально, шаг за шагом)

### Фаза 1 — Движок автономии (Autonomy Engine) 🔴 сердце
**Цель:** Sana сама берёт цель, делает работу руками агента, ревьюит, отчитывается.
1. **Источник целей:** на Trello метка `auto-ok` = карточки, которые Sana вправе брать
   автономно. Поле приоритета + проект (метки уже есть).
2. **Петля автономии:** тик планировщика (расширяем `scheduler_loop`): берёт верхнюю
   `auto-ok` карту из Up Next → маршрутизирует на агента → запускает → по готовности
   двигает карту + пишет отчёт. `max_iterations`, без бесконечных циклов.
3. **Слой диспатча:** `/task`-движок вызывается ВНУТРЕННЕ (не только из Telegram).
   Sana-COO выбирает исполнителя (Maya/Kai/… через Agent tool или claude -p builder).
4. **Ревью-гейт:** результат агента (ветка/PR) → Vex (критика) + Kai (тесты) →
   только потом «готово». Reversible — авто; irreversible/external — спросить директора.
5. **Отчётность:** ежедневный дайджест в Telegram (расширяем бриф): сделано / в работе /
   заблокировано / предложения Sana. + по запросу «итоги дня».
6. **Гард-рейлы:** бюджет токенов/день, лимит параллельных агентов, deny необратимого,
   аудит каждого действия (G5), **kill-switch** (`/stop` — глушит петлю).

**Видим:** дал цель в Trello с меткой `auto-ok` → через время пришёл отчёт «сделано, PR #N, тесты зелёные».

### Фаза 2 — Шина событий (state bus) 🟠
**Цель:** система излучает своё состояние, чтобы дашборд/офис были живыми.
1. Определить lifecycle агента: `spawned → working(task) → in_review → idle → meeting(with) → done → error`.
2. Эмит событий в стор (SQLite/JSONL на хосте движка) — один источник правды о состоянии.
3. «Митинг» = когда агенты реально обмениваются сообщениями (SendMessage/handoff) — логируем тред беседы.

**Видим:** есть файл/таблица, где в реальном времени видно, кто что делает.

### Фаза 3 — Функциональный дашборд 🟠
**Цель:** твой ежедневный экран — без 3D, быстро и по делу.
1. Расширить `sana_web.py`: живая таблица — агент / статус / текущая задача / последний вывод / блок.
2. Читает стор Фазы 2 (polling/SSE).
3. Локально; позже Cloudflare Tunnel если захочешь доступ извне.

**Видим:** открыл страницу — видно пульс компании в реальном времени.

### Фаза 4 — 3D/2D-офис (витрина) 🟢
**Цель:** живой офис как лицо компании.
1. **Стек:** старт с 2D пиксель-офиса (Phaser/PixiJS, паттерн «AI Town») — дешевле, обаятельнее,
   проверено. 3D (Three.js) — апгрейд позже. *(решить — см. открытые вопросы)*
2. Маппинг агент → спрайт: за столом = working; на диване/ТВ = idle; в митинг-руме = meeting.
3. Sana — центральный персонаж (голос/текст) — переиспользуем голосовой пайплайн.
4. Привязка к стору Фазы 2 — офис = живое зеркало, не анимация ради анимации.

**Видим:** открыл сайт — команда «живёт», двигается по реальной работе, Sana в центре отвечает.

### Фаза 5 — Стартовый опыт + полировка 🟢
1. Вход в Windows → авто-открытие: Trello + Telegram + сайт-офис (Планировщик/ярлыки).
2. Раскладка: Sana 3D по центру, «desktop environment» команды слева.
3. Онбординг-режим, звук офиса, мелочи «дорогого» продукта.

## 4. Где это крутится (хостинг движка)
- **VM 1GB** тянет лёгкого 24/7-оператора (ответы, Trello, бриф, Calendar/Gmail), но
  НЕ рой агентов + 3D. → Движок (Фаза 1) и тяжёлые `/task` — на ноуте (когда включён)
  или на более мощном хосте. VM остаётся «всегда на связи» фронтом. *(решить — откр. вопросы)*

## 5. Экономика / токены
- Автономия = непрерывный расход. Claude-подписка имеет rate-limit. →
  **событийная модель** (есть задача → агент проснулся → сделал → уснул), не «10 агентов 24/7».
- Дешёвые задачи → free-модели (Groq/Gemini), сложные → Claude. Дневной бюджет + стоп при превышении.

## 6. Безопасность / доверие (уровни автономии L1–L4)
- L1 ответ/анализ · L2 внутреннее (Trello/файлы/ветка) — авто · L3 PR/драфт — показать · L4 внешнее/необратимое — только «да».
- Автономность заканчивается на PR. Merge/deploy/send — директор. Kill-switch + аудит-лог.
